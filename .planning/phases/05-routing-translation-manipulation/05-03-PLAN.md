  <name>Task 1: Create routing, translation, manipulation API handlers</name>
  <files>src/handler/routing_api.rs, src/handler/translations_api.rs, src/handler/manipulations_api.rs</files>
  <action>
  Follow the exact pattern from trunks_api.rs (use require_config_store! macro, same error response shapes, same StatusCode usage).

  **src/handler/routing_api.rs** — 9 handlers:

  1. `list_routing_tables(State)` — GET /api/v1/routing/tables -> 200 with JSON array
  2. `create_routing_table(State, Json<RoutingTableConfig>)` — POST /api/v1/routing/tables -> validate name not empty, 201 on success
  3. `get_routing_table(State, Path<name>)` — GET /api/v1/routing/tables/{name} -> 200 or 404
  4. `update_routing_table(State, Path<name>, Json<RoutingTableConfig>)` — PUT /api/v1/routing/tables/{name} -> validate, 200 on success, 404 if not found
  5. `delete_routing_table(State, Path<name>)` — DELETE /api/v1/routing/tables/{name} -> 204 on success, 404 if not found
  6. `list_routing_records(State, Path<name>)` — GET /api/v1/routing/tables/{name}/records -> 200 with records array from table, 404 if table not found
  7. `add_routing_record(State, Path<name>, Json<RoutingRecord>)` — POST /api/v1/routing/tables/{name}/records -> load table, append record, save, 201
  8. `delete_routing_record(State, Path<(name, index)>)` — DELETE /api/v1/routing/tables/{name}/records/{index} -> load table, remove record at index, save, 204
  9. `resolve_route(State, Json<RouteResolveRequest>)` — POST /api/v1/routing/resolve -> use RoutingEngine to resolve, return RouteResult as JSON

  For resolve_route, accept a JSON body with `table_name`, `destination_number`, `caller_number` fields. Construct a RouteContext and call engine.resolve(). Return the matched trunk and record info.

  **src/handler/translations_api.rs** — 5 handlers:

  1. `list_translation_classes(State)` — GET /api/v1/translations -> 200
  2. `create_translation_class(State, Json<TranslationClassConfig>)` — POST /api/v1/translations -> validate name, 201
  3. `get_translation_class(State, Path<name>)` — GET /api/v1/translations/{name} -> 200 or 404
  4. `update_translation_class(State, Path<name>, Json<TranslationClassConfig>)` — PUT /api/v1/translations/{name} -> 200 or 404
  5. `delete_translation_class(State, Path<name>)` — DELETE /api/v1/translations/{name} -> 204 or 404

  **src/handler/manipulations_api.rs** — 5 handlers:

  1. `list_manipulation_classes(State)` — GET /api/v1/manipulations -> 200
  2. `create_manipulation_class(State, Json<ManipulationClassConfig>)` — POST /api/v1/manipulations -> validate name, 201
  3. `get_manipulation_class(State, Path<name>)` — GET /api/v1/manipulations/{name} -> 200 or 404
  4. `update_manipulation_class(State, Path<name>, Json<ManipulationClassConfig>)` — PUT /api/v1/manipulations/{name} -> 200 or 404
  5. `delete_manipulation_class(State, Path<name>)` — DELETE /api/v1/manipulations/{name} -> 204 or 404

  Use the same require_config_store! macro pattern. Import it or define a shared version. Each handler follows the same structure: extract state, get config_store, call CRUD method, map result to JSON response.

  For validation: name must not be empty. For routing records, match_type must be valid. Return 400 with descriptive error JSON on validation failure.
