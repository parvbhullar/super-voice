<task type="auto">
  <name>Task 2: Create bridge dispatch functions for WebRTC and WebSocket modes</name>
  <files>src/proxy/bridge.rs, src/proxy/mod.rs</files>
  <action>
Create src/proxy/bridge.rs with two async bridge functions.

1. **dispatch_webrtc_bridge**:

```rust
pub async fn dispatch_webrtc_bridge(
    app_state: AppState,
    session_id: String,
    caller_dialog: DialogStateReceiverGuard,
    caller_sdp: String,
    caller_uri: String,
    callee_uri: String,
    did: &DidConfig,
) -> Result<()>
```

Implementation:
- Create a CancellationToken as child of app_state.token
- Create an event channel (tokio mpsc unbounded) for SessionEvents
- Build an RtcTrackConfig with:
  - mode: TransportMode::WebRtc
  - ice_servers: convert did.routing.webrtc_config.ice_servers (Vec<String>) to Vec<IceServer> using IceServer { urls: vec![url], .. } format, or use a default STUN server ("stun:stun.l.google.com:19302") if none provided
  - ice_lite: did.routing.webrtc_config.ice_lite.unwrap_or(false) via enable_ice_lite
  - codecs: vec![CodecType::Opus, CodecType::PCMU, CodecType::PCMA] (prefer Opus for WebRTC)
  - preferred_codec: Some(CodecType::Opus)
- Create an RtcTrack (the WebRTC target leg) with a unique track_id like "{session_id}-webrtc"
- Call rtc_track.handshake(caller_sdp, Some(Duration::from_secs(10))) to generate the SDP answer
- Create a SIP RtcTrack (the SIP caller leg) with RtcTrackConfig { mode: TransportMode::Rtp, codecs: vec![CodecType::PCMU, CodecType::PCMA], .. } for the SIP side
- Create packet sender/receiver channels for both tracks
- Start both tracks via track.start(event_sender, packet_sender)
- Enter a bridge loop: read AudioFrames from each track's receiver and forward to the other track's send_packet(), respecting cancel_token
- On cancel or either track ending, stop both tracks and return

This is structurally similar to MediaBridge but simplified for the SIP-to-WebRTC case without the full ProxyCallSession machinery.

2. **dispatch_ws_bridge**:

```rust
pub async fn dispatch_ws_bridge(
    app_state: AppState,
    session_id: String,
    caller_dialog: DialogStateReceiverGuard,
    caller_sdp: String,
    caller_uri: String,
    callee_uri: String,
    did: &DidConfig,
) -> Result<()>
```

Implementation:
- Extract ws_config from did.routing.ws_config. Return error if missing.
- Open an outbound WebSocket connection to ws_config.url using tokio_tungstenite::connect_async (the project already uses tokio-tungstenite or tungstenite — check Cargo.toml; if not present, add tokio-tungstenite as dependency).
- Create an mpsc::unbounded_channel for audio bytes (WebsocketBytesSender, WebsocketBytesReceiver).
- Create a WebsocketTrack with the receiver, codec from ws_config.codec (default "pcmu"), ssrc from random u32.
- Spawn a task that reads WS binary messages and sends them into the WebsocketBytesSender channel.
- Spawn a task that reads from the WS write half, receiving outbound audio to send to the remote WS server.
- Create a SIP-side RtcTrack with TransportMode::Rtp for the caller leg.
- Start both tracks with packet channels.
- Bridge loop: forward AudioFrames between SIP track and WS track bidirectionally.
- On cancel or WS close, stop both tracks and close WS connection.

3. Register the module in src/proxy/mod.rs: add `pub mod bridge;`

4. Add unit tests in bridge.rs:
- test_webrtc_bridge_config_defaults: verify RtcTrackConfig is built correctly from a DidConfig with webrtc_bridge mode and default ICE servers
- test_ws_bridge_config_extraction: verify WsBridgeConfig URL and codec extraction from DidConfig
- test_ws_bridge_missing_config_returns_error: verify dispatch_ws_bridge returns error when ws_config is None

Note: Full end-to-end bridge tests require real SIP/WebRTC/WS infrastructure and are not feasible in unit tests. The unit tests validate configuration extraction and error handling. Integration-level verification is in Plan 02.
