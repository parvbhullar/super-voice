1. **src/gateway/mod.rs**: Re-export GatewayManager, GatewayHealthMonitor, and a GatewayStatus struct.

2. **src/gateway/manager.rs**: GatewayManager that tracks gateway state:
   - Internal state: `HashMap<String, GatewayState>` where GatewayState holds:
     - config: GatewayConfig
     - status: GatewayHealthStatus (from runtime_state.rs)
     - consecutive_failures: u32
     - consecutive_successes: u32
     - last_check: Option&lt;Instant&gt;
   - `new(config_store: Arc&lt;ConfigStore&gt;, runtime_state: Arc&lt;RuntimeState&gt;) -> Self`
   - `add_gateway(config: GatewayConfig) -> Result<()>` — validates transport is one of udp/tcp/tls, adds to map, persists via config_store.set_gateway, sets initial health to Active in RuntimeState
   - `remove_gateway(name: &str) -> Result<()>` — removes from map, deletes from config_store
   - `get_gateway(name: &str) -> Option<GatewayInfo>` — returns config + current status (reads from RuntimeState)
   - `list_gateways() -> Vec<GatewayInfo>` — returns all with status
   - `load_from_config_store() -> Result<()>` — loads all GatewayConfigs from Redis, rebuilds internal map, reads health status from RuntimeState for each
   - `record_health_result(name: &str, success: bool) -> Result<()>` — increments consecutive counter, resets the other, checks thresholds:
     - If consecutive_failures >= config.failure_threshold and status is Active: set to Disabled in RuntimeState
     - If consecutive_successes >= config.recovery_threshold and status is Disabled: set to Active in RuntimeState
   - `GatewayInfo` struct: { name, proxy_addr, transport, status, last_check }

   **Important for testability:** Extract the threshold transition logic into a pure function or make `record_health_result` testable without Redis by accepting a trait or allowing GatewayManager to be constructed with mock stores. At minimum, make GatewayState and the threshold check logic available for unit testing (e.g., a `fn check_threshold(state: &mut GatewayState) -> Option<GatewayHealthStatus>` that returns the new status if a transition should occur).

   GatewayManager should be `Arc<Mutex<GatewayManager>>` for thread-safe sharing.

3. Register module in src/lib.rs or wherever modules are declared.

4. **tests/gateway_health_test.rs**: Unit tests for threshold transitions:
   - Create a GatewayState with failure_threshold=3, recovery_threshold=2
   - Test: 3 consecutive failures (record_health_result(false) x3) -> status transitions to Disabled
   - Test: After disabled, 2 consecutive successes -> status transitions back to Active
   - Test: 2 failures then 1 success resets failure counter (so next 2 failures still not enough)
   - Test: 1 failure stays Active (below threshold)
   - Test: 1 success while Disabled stays Disabled (below recovery threshold)

   If GatewayManager requires Redis, test the pure threshold logic function directly. Do NOT require a live Redis connection for these unit tests.

<task type="auto">
  <name>Task 2: GatewayHealthMonitor background task</name>
  <files>src/gateway/health_monitor.rs</files>
  <action>
Create the health monitoring background task:

1. **src/gateway/health_monitor.rs**: GatewayHealthMonitor struct:
   - Holds: Arc&lt;Mutex&lt;GatewayManager&gt;&gt;, CancellationToken
   - `new(manager: Arc&lt;Mutex&lt;GatewayManager&gt;&gt;) -> Self`
   - `start(&self) -> JoinHandle<()>` — spawns a tokio task that:
     - Loops with tokio::select on cancellation token
     - Every iteration: gets list of gateways from manager
     - For each gateway with health check enabled (health_check_interval_secs > 0):
       - Check if enough time has elapsed since last_check (based on health_check_interval_secs)
       - If due: send SIP OPTIONS ping to gateway's proxy_addr
       - OPTIONS ping implementation:
         - For Sofia-SIP gateways (carrier feature): use NuaAgent::send_options() if a shared Sofia agent is available, or create a lightweight SIP OPTIONS via raw UDP/TCP socket as fallback
         - For simplicity in this phase: use a raw SIP OPTIONS request over the configured transport (UDP/TCP/TLS). Build a minimal OPTIONS message per RFC 3261:
           ```
           OPTIONS sip:{proxy_addr} SIP/2.0
           Via: SIP/2.0/{transport} {local_addr};branch=z9hG4bK{random}
           From: <sip:healthcheck@{local_domain}>;tag={random}
           To: <sip:{proxy_addr}>
           Call-ID: {random}@{local_domain}
           CSeq: 1 OPTIONS
           Max-Forwards: 70
           Content-Length: 0
           ```
         - Send via tokio::net::UdpSocket (UDP), tokio::net::TcpStream (TCP), or tokio_rustls (TLS)
         - Wait for response with timeout (5 seconds)
         - Success = any 2xx response received
         - Failure = timeout, connection refused, or non-2xx response
       - Call manager.record_health_result(name, success)
     - Sleep for 1 second between iterations (the per-gateway interval is checked internally)
   - `stop(&self)` — cancels the token

2. The monitor should handle errors gracefully — a failed ping is a health failure, not a crash. Log at warn level on failure, debug on success.

3. For TLS transport OPTIONS: use tokio-rustls with a permissive TLS config (accept self-signed for health checks). If TLS setup fails, count as health check failure.
