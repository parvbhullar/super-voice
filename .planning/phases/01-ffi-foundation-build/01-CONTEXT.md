<decisions>
## Implementation Decisions

### Workspace Structure
- Restructure from single crate `active-call` (v0.3.48) to a Cargo workspace
- New crates in `crates/` directory:
  - `sofia-sip-sys` ‚Äî raw FFI bindings generated by bindgen from nua.h, sdp.h, auth_module.h, su_root.h
  - `sofia-sip` ‚Äî safe Rust wrapper (NuaAgent, NuaHandle, SuRoot, EventLoop, event types)
  - `spandsp-sys` ‚Äî raw FFI bindings generated by bindgen from spandsp.h
  - `spandsp` ‚Äî safe Rust wrapper (DtmfDetector, EchoCanceller, ToneDetector, Plc, FaxEngine)
- Root `active-call` crate depends on the wrapper crates via workspace path dependencies
- `media-gateway` remains a separate repo/crate (not part of this workspace initially)
- Import paths: `use sofia_sip::NuaAgent;`, `use spandsp::DtmfDetector;`

### C Library Sourcing
- **System pkg-config** for both Sofia-SIP and SpanDSP (not vendored)
- `build.rs` in each `-sys` crate uses `pkg-config` crate to discover headers and link flags
  - Sofia-SIP: `pkg_config::probe_library("sofia-sip-ua")` (requires >= 1.13.17)
  - SpanDSP: `pkg_config::probe_library("spandsp")` (requires >= 3.0)
- `bindgen` generates bindings from C headers into `src/bindings.rs` (generated at build time)
- Allowlist specific functions/types to keep binding surface minimal:
  - Sofia-SIP: `nua_*`, `su_root_*`, `sip_*`, `sdp_*`, `nta_*`, `tport_*`
  - SpanDSP: `dtmf_rx_*`, `echo_can_*`, `super_tone_rx_*`, `fax_*`, `t38_terminal_*`, `plc_*`
- All C types treated as opaque (`opaque_type` in bindgen config)
- Docker build installs system packages: `libsofia-sip-ua-dev`, `libspandsp-dev`, `libtiff-dev`, `libjpeg-dev`, `libz-dev`, `clang`, `libclang-dev`
- Runtime image needs: `libsofia-sip-ua0`, `libspandsp3`, `libtiff6`

### Feature Flag Design
- `Cargo.toml` features:
  - `carrier` = ["sofia-sip", "spandsp"] ‚Äî enables C FFI carrier features
  - `minimal` = [] ‚Äî pure Rust, no C dependencies (existing behavior preserved)
  - `offline` = ["ort", "hf-hub", ...] ‚Äî existing offline model support (unchanged)
  - `default` = ["carrier"] ‚Äî carrier features on by default
- Conditional compilation: `#[cfg(feature = "carrier")]` gates all Sofia/SpanDSP code paths
- `build.rs` only runs pkg-config when `carrier` feature is active

### Sofia-SIP‚ÜîTokio Bridge Pattern
- Sofia-SIP has its own event loop (`su_root_t`) ‚Äî single-threaded, callback-based
- Integration pattern: **dedicated OS thread** (not spawn_blocking, because su_root_step needs continuous polling)
  - `std::thread::spawn` runs the Sofia event loop
  - Two `mpsc::unbounded_channel` bridge async Rust ‚Üî Sofia thread:
    - `event_tx/event_rx`: Sofia callbacks ‚Üí Rust async (SofiaEvent enum)
    - `cmd_tx/cmd_rx`: Rust async ‚Üí Sofia commands (SofiaCommand enum)
  - Sofia thread loop: `su_root_step(root, 1)` (1ms timeout) then `cmd_rx.try_recv()` for commands
- C callback trampoline:
  - `extern "C" fn sofia_event_trampoline(...)` ‚Äî the NUA callback
  - Receives `*mut c_void` magic pointer to `event_tx` sender
  - Copies SIP message data from C memory into Rust-owned types
  - Sends via `event_tx.send(SofiaEvent)` ‚Äî non-blocking, unbounded
- Memory safety:
  - All C pointers wrapped in Rust structs with `Drop` impls
  - `NuaAgent` Drop calls `nua_shutdown()` then polls `su_root_step()` until clean
  - `NuaHandle` Drop calls `nua_handle_unref()`
  - `SuRoot` Drop calls `su_root_destroy()`
  - No raw pointer escapes the wrapper crate

### Sofia-SIP Event/Command Types
- `SofiaEvent` enum (sent from C callback to Rust async):
  - `IncomingInvite { handle: SofiaHandle, from: String, to: String, sdp: Option<String> }`
  - `IncomingRegister { handle: SofiaHandle, contact: String }`
  - `InviteResponse { handle: SofiaHandle, status: u16, phrase: String, sdp: Option<String> }`
  - `Terminated { handle: SofiaHandle, reason: String }`
  - `Info { handle: SofiaHandle, content_type: String, body: String }`
- `SofiaCommand` enum (sent from Rust async to C thread):
  - `Respond { handle: SofiaHandle, status: u16, reason: String, sdp: Option<String> }`
  - `Invite { handle: SofiaHandle, uri: String, sdp: String }`
  - `Register { handle: SofiaHandle, registrar: String }`
  - `Bye { handle: SofiaHandle }`
  - `Shutdown`
- `SofiaHandle` wraps `*mut nua_handle_t` with `Clone` (via `nua_handle_ref`) and `Drop` (via `nua_handle_unref`)

### SpanDSP Integration Pattern
- SpanDSP is frame-based (no event loop) ‚Äî much simpler than Sofia-SIP
- Each SpanDSP feature becomes a `Processor` trait impl (existing trait in `src/media/processor.rs`):
  - `DtmfDetector` ‚Äî wraps `dtmf_rx_state_t`, implements `process_frame()`, fires callback on digit detect
  - `EchoCanceller` ‚Äî wraps echo canceller state, processes audio in-place
  - `ToneDetector` ‚Äî wraps `super_tone_rx_state_t`, detects busy/ringback/SIT
  - `PlcProcessor` ‚Äî wraps `plc_state_t`, conceals packet loss
  - `FaxEngine` ‚Äî wraps `fax_state_t` / `t38_terminal_state_t` (more complex, not a simple Processor)
- Registration in StreamEngine:
  - `engine.register_processor("spandsp_dtmf", SpanDspDtmfDetector::create)`
  - `engine.register_processor("spandsp_echo", SpanDspEchoCanceller::create)`
- Per-call memory: ~25-40 KB total for full DTMF + echo + PLC session (negligible)
- All SpanDSP state is per-call (no global state, no threading inside SpanDSP)
- Callback pattern: SpanDSP C callbacks ‚Üí Rust closure via `extern "C"` trampoline + `*mut c_void` user_data

### Docker Build
- Multi-stage Dockerfile:
  - **Builder stage**: `rust:1.82` + system dev packages + `clang`/`libclang-dev` for bindgen
  - **Runtime stage**: `debian:bookworm-slim` + runtime libs only
- Final image size target: ~100-150 MB (runtime libs + binary)
- Binary size target: ~30-40 MB

### Testing Strategy for Phase 1
- Sofia-SIP FFI smoke test: create NuaAgent, start event loop, send/receive a SIP OPTIONS
- SpanDSP FFI smoke test: create DtmfDetector, feed 480Hz+620Hz tone, verify digit callback fires
- Feature flag test: `cargo build --no-default-features` compiles clean without C deps
- Docker test: build image, run binary, verify startup in <1s
- Integration test: Sofia-SIP receives INVITE from rsipstack (proves both stacks can coexist)

### Claude's Discretion
- Exact bindgen allowlist/blocklist tuning
- Whether to generate bindings at build time or check in pre-generated bindings
- Internal buffer sizes for channel bridging
- Exact Dockerfile base image choice (bookworm vs bullseye)
- Whether to use `cc` crate alongside `pkg-config` for any custom C wrapper code

<specifics>
## Specific Ideas

- Sofia-SIP existing Rust crate `sofia-sip` (v0.1.0 on crates.io by iuridiniz) covers MESSAGE only ‚Äî we need to extend to INVITE/REGISTER, so we create our own from scratch
- The `su_root_step(root, 1)` with 1ms timeout is the sweet spot ‚Äî 0ms busy-waits, higher values add latency
- FreeSWITCH's mod_sofia uses the same pattern: profile thread runs su_root_step in a loop, worker thread processes queued events ‚Äî we replicate this in Rust
- SpanDSP's API is frame-based and stateless per-call ‚Äî perfect for the existing `Processor` trait which is `process_frame(&mut self, frame: &mut AudioFrame) -> Result<()>`
- The `opusic-sys` crate already in the project uses the same `-sys` / safe wrapper pattern ‚Äî follow that precedent

</specifics>

- `src/media/processor.rs`: `Processor` trait and `ProcessorChain` ‚Äî SpanDSP processors implement this directly

- `src/media/vad/tiny_silero.rs`: Existing ONNX-based processor ‚Äî model for how SpanDSP processors should integrate

### Established Patterns
- Feature flags gate optional C dependencies: `offline = ["ort", ...]`, `opus = ["opusic-sys"]`
- `build.rs` exists in media-gateway but not in main crate ‚Äî need to add
- Async-first with Tokio: all I/O is async, blocking work goes to `spawn_blocking` or dedicated threads
- Plugin registry: `StreamEngine::register_asr()`, `register_tts()`, `register_vad()` ‚Äî extend for processors

### Integration Points
- `src/app.rs`: `AppStateBuilder::build()` ‚Äî where SIP endpoint is created; Sofia endpoint adds here alongside rsipstack
- `src/media/engine.rs`: `StreamEngine::new()` ‚Äî where processors are registered; add SpanDSP factories
- `src/config.rs`: `Config` struct ‚Äî needs new fields for carrier features (sip_stack selection, dsp toggles)
- `Cargo.toml` root: restructure to `[workspace]` with `members = ["crates/*"]`
