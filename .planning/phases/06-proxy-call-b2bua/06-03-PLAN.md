The sip_proxy branch must be inserted BEFORE the invitation_handler creation (around line 288). The logic:
1. Extract the called number from the INVITE Request-URI
2. Look up the DID via `config_store.get_did(&called_number)`
3. If DID exists and `did.routing.mode == "sip_proxy"`: call `dispatch_proxy_call(...)` and `continue` (skip the normal invitation_handler path)
4. Otherwise: fall through to existing invitation_handler logic

This requires access to `self.config_store` (available on AppState) and the `tx.original.uri` for the called number.

1. **Route resolution**: Look up routing table for the DID's trunk. Create RoutingEngine from config_store. Build RouteContext from caller/callee URIs. Call resolve() to get RouteResult. If no route found, respond 404 to caller and return.

2. **Load trunk config**: Get TrunkConfig from config_store by RouteResult.trunk name. If not found, respond 503.

5. **Build ProxyCallContext**: session_id, caller, callee, trunk name, DID number, routing table name, start_time=Utc::now().

6. **Create ProxyCallSession**: Pass context, cancel_token (child of app_state.token), caller_dialog, dialog_layer, config_store.

7. **Register in active calls**: Insert into app_state.active_calls with session_id. Create a guard that removes on drop.

8. **Run session**: Call session.run(trunk, caller_sdp, caller_uri, callee_uri). On completion (success or error), the guard drops and removes from active_calls.

9. **Spawn as tokio task**: The entire dispatch should be spawned so it doesn't block the SIP listener.

Extend `src/proxy/session.rs`:

**Hold/Resume detection**:
- Add `fn parse_sdp_direction(sdp: &str) -> SdpDirection` that scans SDP for a=sendonly, a=recvonly, a=inactive, a=sendrecv. Returns enum `SdpDirection { SendRecv, SendOnly, RecvOnly, Inactive }`.
- In the bridge_loop, when a re-INVITE arrives (DialogState with new SDP):
  - Parse SDP direction
  - If SendOnly or RecvOnly or Inactive: set phase to OnHold, emit HoldDetected, call leg_a.suppress_forwarding() (or leg_b depending on which side sent it)
  - If SendRecv (after hold): set phase to Bridged, emit ResumeDetected, call resume_forwarding()
  - Forward the re-INVITE to the other leg (relay SDP)

**REFER handling (call transfer)**:
- Add `async fn handle_refer(&mut self, refer_uri: &str, callee_dialog: &mut DialogStateReceiverGuard) -> Result<()>`:
  1. Set phase to Transferring
  2. Emit TransferInitiated { target: refer_uri }
  3. Create a new outbound INVITE to refer_uri via FailoverLoop (reusing the trunk config)
  4. If new leg connects:
     a. Hang up old callee dialog
     b. Replace callee_dialog with new dialog
     c. Create new MediaBridge between caller and new callee
     d. Set phase to Bridged
  5. If new leg fails: keep existing call, log warning, set phase back to Bridged

**MediaBridge wiring** (extend the run method from Plan 02):
- After FailoverResult::Connected, before entering bridge_loop:
  1. Create RtcTrack for caller leg, RtcTrack for callee leg
  2. Create VoiceEnginePeer for each wrapping a MediaStream
  3. Call optimize_codecs() to pick best common codec
  4. Create MediaBridge with the two peers
  5. Call media_bridge.start()
  6. Pass media_bridge reference into bridge_loop for hold/resume control

Write tests for parse_sdp_direction as unit tests (pure function, easy to test). The hold/resume integration and REFER will be tested in Plan 05 integration tests.
