  <action>
Create `src/proxy/failover.rs`:

**FailoverResult enum**:
- Connected { gateway_addr: String, dialog_guard: DialogStateReceiverGuard, sdp: Option<String> }
- NoFailover { code: u16, reason: String } (stopped by nofailover code)
- Exhausted { last_code: u16, last_reason: String } (all routes tried)
- NoRoutes

**FailoverLoop struct**:
- Fields: dialog_layer: Arc<DialogLayer>, config_store: Arc<ConfigStore>, cancel_token: CancellationToken
- `pub async fn try_routes(&self, trunk: &TrunkConfig, caller_sdp: &str, caller_uri: &str, callee_uri: &str) -> Result<FailoverResult>`:
  1. Load gateway configs for trunk.gateways from config_store
  2. For each gateway (respecting weight/priority ordering):
     a. Create outbound INVITE via dialog_layer with InviteOption (to callee_uri via gateway.proxy_addr, sdp=caller_sdp)
     b. Create a new DialogStateReceiverGuard for the UAC dialog
     c. Wait for dialog state transitions:
        - Early { response } with SDP: store as early_media_sdp, continue waiting
        - Confirmed { response } with SDP: return Connected with SDP (or use early_media_sdp if 200 OK body empty)
        - Terminated with status code: check nofailover_sip_codes — if code is in list, return NoFailover; else try next gateway
     d. Timeout after 30 seconds (configurable): try next gateway
  3. If all gateways exhausted, return Exhausted

The early media storage is critical: if 183 comes with SDP but 200 OK has empty body, use the 183 SDP as the answer SDP. This handles carriers that only send SDP in 183.

Unit tests should use a mock ConfigStore pattern or test the nofailover code checking logic as a pure function extracted from the loop. The `is_nofailover(code, &trunk)` helper should be a standalone `pub fn`.

  <action>
Create `src/proxy/session.rs` with `ProxyCallSession`:

**ProxyCallSession struct**:
- Fields:
  - context: ProxyCallContext (from types.rs)
  - cancel_token: CancellationToken
  - caller_dialog: DialogStateReceiverGuard (UAS — the inbound leg)
  - phase: Arc<RwLock<ProxyCallPhase>> (current call phase)
  - dialog_layer: Arc<DialogLayer>
  - config_store: Arc<ConfigStore>
  - event_tx: tokio::sync::mpsc::UnboundedSender<ProxyCallEvent>
  - early_media_sdp: Option<String> (stored from 183)
  - answer_sdp: Option<String>

**Key methods**:
- `pub fn new(context, cancel_token, caller_dialog, dialog_layer, config_store) -> (Self, mpsc::UnboundedReceiver<ProxyCallEvent>)`: Creates session and returns event receiver
- `pub async fn run(&mut self, trunk: &TrunkConfig, caller_sdp: &str, caller_uri: &str, callee_uri: &str) -> Result<()>`:
  1. Set phase to Ringing
  2. Create FailoverLoop and call try_routes
  3. On FailoverResult::Connected:
     a. Set phase to Bridged
     b. Emit ProxyCallEvent::Answered { sdp }
     c. Enter the bridge loop (monitor both dialog guards for state changes)
  4. On early media during failover:
     a. Set phase to EarlyMedia
     b. Emit ProxyCallEvent::EarlyMedia { sdp }
     c. Store sdp as early_media_sdp
  5. On failure: set phase to Failed, emit Terminated

- `async fn bridge_loop(&mut self, callee_dialog: DialogStateReceiverGuard) -> Result<()>`:
  Monitor both caller_dialog and callee_dialog via tokio::select!:
  - caller Terminated -> hang up callee, emit Terminated, break
  - callee Terminated -> hang up caller, emit Terminated, break
  - cancel_token cancelled -> hang up both, break
  This is the call state machine once media is bridged.

- `pub fn phase(&self) -> ProxyCallPhase`: returns current phase
- `pub fn session_id(&self) -> &str`
- `pub fn context(&self) -> &ProxyCallContext`

Add `pub mod session; pub mod failover;` to src/proxy/mod.rs (alongside the modules from Plan 01).

Note: The actual MediaBridge wiring (connecting RtcTracks) will happen in Plan 03 when we integrate with the call dispatch path. This plan focuses on the SIP dialog state machine and failover logic.
